HTTP协议

HTTP协议

1.概述

  • HTTP，超文本传输协议，是一种应用层协议
  • Web - www 三个要素
    • html：页面展示所使用的语言
    • URL 页面位置
    • HTTP/HTTPS 使用的传输协议

URL统一资源定位符，URI统一资源标识符

URL是URI的子集，URL是URI的一种方法

举例：一个人的URI就是能唯一可以标识该人的身份，可以是身份证号，可以是他的住址加上他的姓名

  • HTTP是基于IP协议和TCP协议的
  • 在使用域名的情况下还要先使用DNS协议（应用层协议）

2.报文

  • 报文的纯文本的，会被转换成以ASCII码格式传输

  • 请求报文由四部分组成

    • 请求行
    • 请求头
    • 换行（CRLF）
    • 消息体

  • 响应报文也由四部分组成

    • 状态行
    • 响应头
    • 换行（CRLF）
    • 消息体

请求报文

  • 请求行
    • 请求方法
      • GET / POST / PUT /DELETE / HEAD
    • URL
      • 结合请求头Host属性可以使用相对路径
    • HTTP版本
  • 请求头
    • Accept：告诉服务端，客户端浏览器可以处理（接收）什么类型的响应
      • text/html
      • img/png
      • img/*（一种类别所有类型）
      • */ *（所有类别所有类型）
    • Accept-Encoding：能接收的编码
    • Connection：连接状态
    • User-Agent 作用：指明客户端使用的操作系统、浏览器版本和名称
    • Content-Type：消息体的类型

3.HTTP各版本

一共有四个版本

  • HTTP 0.9
  • HTTP 1.0
  • HTTP1.1
  • HTTP2

HTTP3即将到来，放弃了TCP协议，而是用UDP协议的某种实现，这里不再多述

HTTP 0.9

  • 只支持GET方法
  • 没有请求头概念
  • 服务端响应后立即关闭TCP连接
  • 传输内容只支持html文本传输

​ GET /index.html

HTTP 1.0

  • 增加了POST / PUT / DELETE / HEAD 方法，状态码等
  • 可以指定HTTP版本号
  • 增加了请求头和响应头的概念
  • 传输内容增加了图片，音频等
  • 服务端响应后也是立即关闭TCP连接，如果要长连接则需要设置Connection属性

HTTP 1.1

  • 长连接，新增请求头的Connection属性，属性值为keep-alive表示为长连接（HTTP 1.1默认），close表示断开（HTTP 1.0默认）

长连接可以通过设置Keep-Alive属性来设置超时时间以及最大请求数量。

这里的时间如果超过了TCP的KeepAlive时间（如果已经开启）会以TCP为准

大部分应用都会实现心跳包功能，客户端向服务端发送心跳

  • 管道化：可以不等待前一个请求返回，发送下一个请求。但是响应顺序还是和请求顺序一致
  • 缓存处理：新增字段cache-control

当浏览器请求资源时，先看是否有缓存的资源，如果有缓存，直接取，不会再发请求，如果没有缓存，则发送请求。 通过设置字段cache-control来控制缓存。

  • 断点传输

在上传/下载资源时，如果资源过大，将其分割为多个部分，分别上传/下载，如果遇到网络故障，可以从已经上传/下载好的地方继续请求，不用从头开始，提高效率

HTTP 2

  • 二进制分帧

Http 2以前采用文本协议

  • 多路复用

所有请求都是通过一个TCP连接完成，与管道不同，响应的顺序不必按照请求顺序返回

  • 头部压缩

一般而言消息体都会经过压缩（gzip或者本身就是二进制），而头部是不压缩的，是文本文件类型。而随着头部的内容越来越多，为了加快传输效率，HTTP 2把头部也进行了压缩

4.GET和POST，HEAD的区别

POST

  • 最简单的区别：GET把请求参数url中而POST放在消息体重
  • GET请求结果会被浏览器主动缓存，而POST不会除非主动设置
  • 然后GET在浏览器回退时会从之前的缓存拿到结果，POST会再次提交请求
  • GET请求的URL可以作为书签而POST不行
  • GET只支持URL编码而POST可支持多种编码方式

HEAD

  • 不会返回消息体，只返回头部
  • 一般用于
    • 只请求资源首部
    • 检测超链接的有效性
    • 检查网页是否被修改

5.HTTP状态码

  • 1开头表示请求已经接受需要进一步处理（临时响应）
    • 100：已经收到一部分请求，正在等待其他部分。请求者应该继续执行操作
    • 101：需要更换协议，服务器已经确认并准备切换
  • 2开头表示成功接收了请求的状态码
    • 200：服务器已经处理了请求
    • 201：已创建
    • 202：已接收（可能尚未处理）
  • 3开头表示要完成请求需要进一步操作（重定向）
    • 301：永久移动，URL已经移动到新的位置，会自动将请求者转到新的位置
    • 302：临时移动
  • 4开头表示客户端可能发生了错误，妨碍了服务端的处理
    • 400：URL请求语义有误，不能被服务器理解
    • 401：没有身份认证或者身份认证有误，无权访问
    • 402：保留暂未使用，本意是留给数字支付
    • 403：表示客户端完成了身份认证，但是并没有响应的权限访问该资源
    • 404：路劲不对，服务端找不到对应的资源
  • 以5开头表示服务端可能内部出现了错误，无法处理请求
    • 500：服务端内部错误
    • 501：没有能力处理请求，可能是没有能力识别该请求方法
    • 502：错误网关。可能没有启动tomcat（没有启动服务器）
    • 503：目前服务器无法使用（可能是维护或者过载）可以返回延迟时间
    • 504：网关超时，服务器作为网关未能从上游服务器获得消息（可能是配置错误）

6.HTTPS协议

  • 在HTTP的基础上加SSL层加密
  • SSL版本
    • TLS协议是SSL3.1版本
    • SSL1.0到3.0版本都是不安全的

加密过程

  • 加密过程使用了对称加密以及非对称加密两种加密方式
    • 加密内容使用对称加密
    • 证书验证阶段用的是非对称加密

对称加密：加密解密用的同一个钥匙，钥匙的传输的安全性存在隐患

非对称加密：公钥加密的信息只要私钥才能解开，反之亦然。（RSA算法）

服务端有私钥，将公钥发给浏览器就是非对称加密，但是公钥可能被截取，服务端向浏览器发的消息并不安全，浏览器发给服务端的会安全些（其实也不安全因为公钥不一定是服务器的，可能被替换）。

同理使用（两组公钥和私钥）或者（公钥私钥以及公钥加密过的对称秘钥）都是不安全的，因为都无法确保公钥是否被替换

公钥被替换这种攻击叫做中间人攻击

只要确保了公钥是属于服务器的就能保住安全型

  • 这里我们需要一个第三方机构（CA）来保证公钥的真实性
  • 操作系统浏览器里面是有权威机构（CA）的根证书（CA公钥信息）的，逐层嵌套，相互保证可靠性。

数字证书

  • 加密过程：首先服务端（网站）需要向CA机构申请证书，CA机构拥有这些权威的公钥以及自己的私钥。CA用私钥对网站的公钥 、域名等信息的哈希值进行加密（数字签名）然后和明文信息（网站的公钥 、域名等信息）共同组成了数字证书。
  • 解密过程：首先有一个根证书，根证书里面的公钥可以解开下一个层证书的信息得到下一层的公钥，以此类推。然后得到了CA的公钥对证书的加密部分解密得到哈希值，对明文信息的使用哈希算法（哈希不可逆）得到哈希值与之对比，可以发现证书是否是服务器的，从而可以使用明文里的公钥了。

浏览器的私钥并没有参与数字签名的加密和解密过程

7.输入一个URL发生了什么

  • DNS解析
  • 客户端和服务端的TCP连接
  • 客户端发送HTTP请求
  • 服务端响应请求返回报文
  • 浏览器解析渲染页面
  • 连接结束
